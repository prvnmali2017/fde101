# The problem with raw LLMs

- LLMs don't know your customer's data
- They hallucinate confidently
- They can't take actions on their own

<!-- Frame the two fixes: RAG (give it knowledge) and tools/agents (give it actions). The rest of the chapter is these two. -->


---

# What is RAG?

- Retrieval-Augmented Generation
- Fetch relevant docs → put in prompt → LLM answers grounded
- Answers cite real sources

<!-- Reference `../docs/vector-dbs-and-rag-architectures.md` as the deep-dive companion. RAG = open-book exam for the LLM. -->


---

# RAG vs fine-tuning

| | RAG | Fine-tuning |
|--|-----|-------------|
| Update data | Re-index (minutes) | Retrain (expensive) |
| Citations | Yes | No |
| Best for | Changing knowledge | Changing behavior/style |

<!-- FDE default is RAG — customer docs change constantly and citations build trust. Fine-tuning is rare in pilots. -->


---

# The RAG pipeline

```
Docs → Chunk → Embed → Vector DB
Query → Embed → Search → Prompt+context → LLM → Answer
```

<!-- Walk each step. This maps to `../lab/src/ingest/rag.py`. Tell students we'll build ingestion, then retrieval. -->


---

# Embeddings explained

- Text → vector of numbers capturing meaning
- Similar meaning = close vectors
- Search = nearest neighbors to the query vector

<!-- Keep intuitive. "King - man + woman ≈ queen" style. The model converts text to coordinates; we find nearby coordinates. -->


---

# Chunking strategies

- Fixed window (256–512 tokens, overlap)
- Paragraph/section based (the lab)
- Semantic / parent-child (advanced)

<!-- Show `_chunk_text()` in the lab. Chunking quality drives answer quality as much as the model. Too big = noisy; too small = lost context. -->


---

# Vector database landscape

- ChromaDB (local/dev — the lab)
- pgvector (already-on-Postgres)
- OpenSearch / Bedrock KB (AWS production)
- Pinecone / Weaviate / Qdrant (SaaS/self-host)

<!-- Reference the full matrix in `../docs/vector-dbs-and-rag-architectures.md`. For pilots: Chroma or Bedrock KB. For prod: OpenSearch or pgvector. -->


---

# Choosing a vector DB (FDE decision)

- "Data must stay in our AWS?" → OpenSearch / Bedrock KB
- "Already on Postgres?" → pgvector
- "SaaS ok, move fast?" → Pinecone
- "Local POC?" → ChromaDB

<!-- This decision tree is gold in interviews and real engagements. Tie each to a customer constraint, not a preference. -->

# Building ingestion

- Load docs → chunk → add to collection
- Store text + metadata (source!)
- Metadata enables citations + filters

<!-- Show `ingest_documents()` in `../lab/src/ingest/rag.py`. Note we store `source` so answers can cite the doc. -->


---

# Retrieval

- Embed the query
- `collection.query(..., n_results=3)`
- Return chunks + sources + scores

<!-- Show `search_documents()`. Top-k retrieval. Discuss tuning k: too low misses context, too high adds noise/cost. -->


---

# Assembling the prompt

- System prompt sets rules (answer only from context)
- Inject retrieved chunks as context
- Instruct: cite sources, say "I don't know" if absent

<!-- Show `SYSTEM_PROMPT` in `../lab/src/agent/prompts.py`. The "only answer from context" rule is your anti-hallucination guardrail. -->


---

# Prompt engineering for reliability

- Be explicit about boundaries
- Demand citations
- Low temperature for factual tasks
- Tell it what to do when unsure

<!-- Reliability > cleverness in enterprise. A boring, predictable assistant that cites sources beats a creative one that guesses. -->


---

# From RAG to Agents

- RAG = knowledge
- Agent = knowledge + actions
- Agent decides WHEN to call tools

# Tool use / function calling

- Define tools with JSON schema
- LLM emits a tool call → you run it → feed result back
- LLM composes final answer
